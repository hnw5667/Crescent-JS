Deployment

From your laptop to the world — run the app, serve the API, render SSR, and ship to production in a few commands.

CLI · API server · SSR
ℹ️
This guide covers the deployment patterns built into crescent.js. The framework ships with a crescent run CLI, a built-in API server (api_make), and server-side rendering (render_full_page) — so you have everything you need without extra tooling.

Running Your App

The fastest way to run a Crescent.js app is the crescent CLI. Pick a port, or let it default to 3000.

bash
crescent run app.js 3030   # Run on a specific port
crescent run app.js        # Auto-runs on port 3000

A Production-Ready Entry Point

A typical app.js registers pages, sets up the API server, and starts listening. Combine the renderer, API server, and database in one process:

javascript
const crescent = require('crescent-js');

// --- Frontend: register pages with the renderer
crescent.renderer.register_page(homePage);
crescent.renderer.register_page(aboutPage);
crescent.renderer.basehtml('<html><body><div id="rocket-root"></div></body></html>');

// --- Backend: create the API server
const api = crescent.api_make({
  api_id: 'prod_api',
  port: process.env.PORT || 3000,
  host: '0.0.0.0',        // bind to all interfaces in production
  cors: true
});

api.add_endpoint('GET', '/', (req, res) => {
  // SSR: serve fully rendered HTML
  const html = crescent.renderer.render_full_page('home');
  res.writeHead(200, { 'Content-Type': 'text/html' });
  res.end(html);
});

// --- Start
await api.start();
console.log(`🚀 crescent.js running on port ${process.env.PORT || 3000}`);
🌍
Bind to 0.0.0.0 in production. The default localhost only accepts local connections — most hosts (Render, Railway, Fly.io, a VPS) need 0.0.0.0 to route traffic in.

Environment Variables

Keep secrets and config out of your code. Read them from process.env:

bash
# .env
PORT=3000
NODE_ENV=production
COOKIE_SECRET=replace-with-a-long-random-string
DATABASE_URL=optional-if-using-external-db
javascript
const login = crescent.auth.login({
  collection: 'users',
  cookie: {
    secret: process.env.COOKIE_SECRET,   // never hardcode this
    secure: process.env.NODE_ENV === 'production',
    same_site: 'lax'
  }
});

Server-Side Rendering (SSR)

Use render_full_page to return complete HTML from the server — great for SEO and fast first paint:

javascript
api.add_endpoint('GET', '/about', (req, res) => {
  const html = crescent.renderer.render_full_page('about');
  res.writeHead(200, { 'Content-Type': 'text/html' });
  res.end(html);
});

For client-side navigation between registered pages, use the renderer directly:

javascript
crescent.renderer.navigate('home');   // swap to the home page in-place

Deploying to a Host

Any Node.js host works. The general recipe:

  1. Push your code to a Git repository
  2. Let the host run npm install
  3. Set the start command to crescent run app.js $PORT (or node app.js if you start the server yourself)
  4. Set your environment variables (PORT, COOKIE_SECRET, NODE_ENV=production)
▲

Vercel / Netlify

Great for SSR functions. Set the build to your entry and provide env vars in the dashboard.

🚂

Railway / Render

Long-running Node processes. Use 0.0.0.0 and the platform's PORT.

🖥️

VPS / Docker

Full control. Run with a process manager (PM2) and put a reverse proxy (Nginx/Caddy) in front.

Docker

dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .
ENV NODE_ENV=production
EXPOSE 3000
CMD ["node", "app.js"]

PM2 (process manager)

bash
npm install -g pm2
pm2 start app.js --name crescent-app --env production
pm2 save
pm2 startup     # auto-restart on reboot

Production Checklist

✅
Before you ship, double-check:
  • Cookie secret is set to a long random value (not the default)
  • secure: true on cookies (requires HTTPS)
  • API server binds to 0.0.0.0, not localhost
  • NODE_ENV=production is set
  • Strong passwords enforced (default rejects weak)
  • http_only: true stays on to block XSS cookie access
